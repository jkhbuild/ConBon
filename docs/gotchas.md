# Stack gotchas

Per-phase non-obvious findings that bit during implementation. Pointer from project [CLAUDE.md](../CLAUDE.md). Each bullet is generalizable — if the same stack ships in a future project, expect to hit the same trap.

## Phase 2 (Prisma 7, Postgres, Next 16)

- **Prisma 7 dropped `datasource.url`.** Validation error "datasource property `url` is no longer supported" means the URL goes in `prisma.config.ts` via `env("DATABASE_URL")` plus `import "dotenv/config"` at the top of that file. `env()` does NOT auto-load `.env`.
- **`prisma migrate diff` flag renamed.** `--from/to-schema-datamodel` is gone; use `--from-schema` / `--to-schema`.
- **Partial indexes need raw SQL.** Generate the base migration with `prisma migrate diff --from-empty --to-schema prisma/schema.prisma --script`, append the `CREATE INDEX ... WHERE ...` to the migration file, then apply.
- **`prisma db seed` reads `prisma.config.ts` `migrations.seed`**, not the old `package.json` `prisma.seed` field.

## Phase 4 (Next 16, next/font, eslint-config-next)

- **`next/font/google` requires network at build time.** It downloads + subsets the fonts during `next build`; offline machines fail. Cache survives across rebuilds. If a CI runner is air-gapped, swap to `next/font/local` and commit the woff2.
- **Pre-paint script needs `<html suppressHydrationWarning>`.** React would otherwise diff the server-rendered `<html>` (no `data-theme`) against the client DOM that the inline script has already mutated, and log a hydration warning. The suppression is scoped to the single attribute via React; it does not silence content mismatches inside children.
- **`eslint-config-next` doesn't ship `react/no-danger` or `react-hooks/exhaustive-deps` rule defs out of the box.** `eslint-disable-next-line react/no-danger` or `react-hooks/exhaustive-deps` directives fail lint with "Definition for rule ... was not found." Either drop the directive (preferred when the underlying rule isn't actually firing) or install the corresponding plugin. Project currently has neither installed.

## Phase 5 (@dnd-kit, React 19, Next 16)

- **`DndContext` auto-IDs hydration-mismatch on every card.** Symptom: `aria-describedby="DndDescribedBy-0"` (server) vs `DndDescribedBy-1` (client). Root cause: @dnd-kit uses a module-level counter for the announcer id, and the counter increments at different points during SSR vs CSR. Fix: pass `id="conbon-board"` (or any stable string) to `DndContext`. The id replaces the auto-generated counter entirely.
- **React 19 types `inert` as `boolean`, not the HTML5 boolean attribute.** `inert={true}` works; the old workaround `<div {...{ inert: "" }}>` errors with "Type 'string' is not assignable to type 'boolean | undefined'." Bare `inert={isDragging}` is the correct spelling under `@types/react@19`.
- **CSS custom-property style keys need a cast in `CSSProperties`.** TS's `CSSProperties` type doesn't accept `"--p-color"` as a key; cast via `["--p-color" as string]: ...` inside the style object literal. Alternative is `as React.CSSProperties` on the whole object, but the per-key cast keeps the rest of the object type-checked.
- **`document.body.style.userSelect = "none"` during drag, restored on cleanup.** Centralizing this in `Board`'s `useEffect` keyed off `draggingCardId` is cleaner than adding a per-card CSS rule that has to be conditioned on a drag-state class. Matches /web-design-guidelines guidance.
- **`closestCorners` is the right collision detector for kanban-style boards.** Tall columns + variable-height cards confuse `closestCenter` (a card dropped near a column edge can register against the wrong column's centroid). Corners-based detection picks the visually closest target.

## Phase 6 (Radix UI, tRPC v11 mutations, React 19 event delegation)

- **Radix Dialog renders Overlay and Content as siblings under Portal.** The prototype's CSS nested `.modal` inside `.modal-backdrop` and used `place-items: center` on the backdrop to center the modal. Radix's structure is `<Portal><Overlay/><Content/></Portal>` — they're peers. The fix is to reposition `.modal` as `position: fixed; top: 50%; left: 50%; transform: translate(-50%, -50%)`. The backdrop's centering CSS becomes inert (no children) but causes no harm.
- **Radix ContextMenu.Trigger inline-wraps its children.** Default Trigger element is a span (or div, depending on primitive); inside a flex / grid parent that span becomes the layout child and the actual content loses its layout role (e.g. `.card` no longer behaves as a flex item). Fix: `<ContextMenu.Trigger style={{ display: "contents" }}>` — the wrapper drops out of layout and the real child becomes the layout child again. Right-click still bubbles up to the wrapper element.
- **Mixing custom buttons with menu Items in a horizontal row.** Default Radix menu primitives auto-close on `onSelect` (Item) but not on plain button clicks. For a horizontal row of choices (e.g. priority dots 1-5) where you want clicking any one to close the menu + fire its action, wrap each button with `<ContextMenu.Item asChild>`. Roving focus is vertical so arrow keys behave a bit oddly across the row, but the click + Enter paths work and the menu closes — better than the alternative of a non-Item click leaving the menu open.
- **Distinguishing click from drag on the same draggable element.** `useSortable` listeners include `onPointerDown` that participates in @dnd-kit's activation; a naive `onClick` on the same element will fire after both real clicks AND after a drag where the user happens to release on the source. Pattern: snapshot `clientX/Y` on `pointerdown` via a composed listener (spread `listeners` first, then override `onPointerDown` with one that forwards to `listeners.onPointerDown` after capturing position), and in `onClick` only fire the click action if the pointer moved less than the PointerSensor's `activationConstraint.distance` (5px). The two paths become exclusive: <5px = click, ≥5px = drag.
- **tRPC v11 optimistic-update primitive shape.** The minimum lifecycle is `cancel()` → `getData()` → `setData(undefined, updater)` → return previous → on error, `setData(undefined, previous)` → on settle, `invalidate()`. `cancel()` matters: an in-flight refetch can overwrite the optimistic patch right after it lands. The `undefined` first arg to `setData` is the query input shape (for input-less queries like `cards.list`). A generic helper that takes the list-util object structurally + a patch fn cleans this up across mutations (see `lib/hooks/useOptimisticListMutation.ts`).
- **Don't optimistic-insert when you don't have the real ID.** `cards.create` could optimistically prepend a temp-ID row to `cards.list`, then swap to the real row when the mutation resolves. That swap logic plus reconciliation with intervening cache invalidations is more code than the underlying mutation. Cheaper: just `invalidate()` on settle — a sub-50ms refetch reconciles. Same logic for `cards.restore` (touches two lists, no clean optimistic story).
- **`preview_click` does not always fire React 19's onClick.** Confirmed during Phase 6 verification: filling an input via `preview_fill` then `preview_click('button.btn-primary')` did NOT trigger React's `onClick` on a Radix Dialog button, but calling `saveBtn.click()` (native HTMLElement.click) via `preview_eval` did. This is a tooling quirk — the underlying React component works for real users (whose clicks go through the browser's full trusted-event pipeline). Worth checking via `.click()` in eval before suspecting the component.

## Phase 7 (Next 16 proxy, Auth.js v5, JWT sessions)

- **`middleware.ts` is deprecated in Next 16 — use `proxy.ts`.** Same `export default` + `export const config` shape, same matcher syntax, same `req.nextUrl` API. `middleware.ts` still works but emits `The "middleware" file convention is deprecated. Please use "proxy" instead.` on build. Migrate by file-rename (no code change inside).
- **`NEXTAUTH_URL` + autoPort + `trustHost: true` interaction.** When `next dev`'s autoPort falls back off port 3000 (because something else is on it) and `NEXTAUTH_URL` is pinned to `http://localhost:3000`, Auth.js v5 builds the proxy's redirect target off `NEXTAUTH_URL` — sending the browser to the wrong host even though `trustHost: true` is set. Fix: leave `NEXTAUTH_URL` unset in dev (Auth.js then uses the request's Host header), and set it only in prod where the URL is stable.
- **`/api/trpc/*` must be excluded from the proxy matcher.** Otherwise an expired-session tRPC request gets a 307 → `/signin` (HTML) instead of a JSON 401, which the tRPC client can't parse — it surfaces as an opaque "fetch failed" instead of `UNAUTHORIZED`. The procedure-level guards (`protectedProcedure` / `adminProcedure` / `managerProcedure`) handle the unauth case correctly, so the proxy is redundant for the tRPC route. Same pattern will apply to Phase 8's `/api/events` (SSE) and `/api/auth/*` (Auth.js's own handlers).
- **JWT role updates require sign-out + sign-in to take effect.** With `session: { strategy: "jwt" }` and an `if (!token.userId)` guard in the `jwt` callback, the role is stamped once on first sign-in and stays in the token. Updating `Person.role` directly in DB doesn't flow to existing sessions. If Phase 9's "promote to Admin" flow needs immediate effect, either drop the guard (re-query Postgres on every request) or call `signIn({ redirect: false })` from the admin UI to force a token refresh. At 4–8 users the slight propagation lag (until next sign-in) is acceptable.
- **Auth.js v5 middleware/proxy callback shape.** Pattern is `export default auth((req) => { ... })` where `req.auth` is the resolved `Session | null`. The `auth` import comes from your local `auth.ts` (returned by `NextAuth({...})`). v4's `withAuth` HOC pattern is gone.
- **`@auth/prisma-adapter` is optional in v5.** With Person already serving as the canonical user table, JWT session strategy + a `signIn` callback that does the Person upsert is simpler than adding User/Account/Session/VerificationToken models. The adapter's value is around OAuth account-linking (one Person, multiple Google accounts), which doesn't matter for an internal app where one email = one Person. Default to JWT unless that constraint changes.
- **Next 16 `instrumentation.ts` runs once per Node process and gates on `NEXT_RUNTIME === "nodejs"`.** It also fires during `next build` (data collection / static analysis). Bootstrap code that touches Postgres must be wrapped in a `try/catch` so an unreachable DB during build doesn't crash the build pipeline. The `bootstrapped` boolean guard inside the module avoids re-running on hot reload within a single dev process.
- **Dev-only `CredentialsProvider` for bypass routes through the same `signIn` / `jwt` / `session` pipeline as Google.** Cheaper than a context-layer override in tRPC and exercises the real allowlist + Person-upsert code path. Gate on `NODE_ENV !== "production"` and only push the provider when `AUTH_DEV_USER_EMAIL` is set. The `authorize` callback returns `{ email, name }` — no password — and Auth.js packs that into the JWT.

## Phase 8 (Postgres LISTEN/NOTIFY, SSE in Next 16, EventSource reconnect)

- **`LISTEN` needs a dedicated `pg.Client`, not the Prisma pool.** `PrismaPg`'s adapter manages a connection pool that may rotate or close any connection; a `LISTEN` registration lives on the connection that issued it, so the moment Prisma rotates that connection out the channel goes silent — no error, just no events. Open one dedicated `Client` and pin it for the process. Verify with `SELECT pid, query FROM pg_stat_activity WHERE query = 'LISTEN conbon_events'` — should always be exactly one row.
- **`globalThis` singleton survives Next dev hot reload.** A module-level `const client = new Client(...)` evaluates fresh on every file save, leaking the previous LISTEN connection on each hot reload. Same fix `lib/db.ts` uses for the Prisma client: `(globalThis as any).__conbonListener ??= { client: null, subscribers: new Set(), starting: null }`. Combined with lazy startup (only connect on first subscriber) it also avoids opening a stray connection during `next build`'s static-analysis pass.
- **`pg_notify` payloads must stay under 8000 bytes.** Postgres enforces an atomic-delivery byte limit; pushing full row bodies through NOTIFY blows past it on rows with long text fields and silently fails. Keep the payload to what the client needs to choose an invalidation target (table + op + id) and let the client re-fetch the authoritative list via tRPC. The client wasn't subscribing for data — it was subscribing for "something changed, refetch."
- **Trigger function: `NEW` is null on `DELETE`.** Reading `NEW.id` inside a DELETE-firing trigger errors at runtime. Branch on `TG_OP` and pull from `OLD` for DELETE, `NEW` otherwise. Symmetric on the `RETURN` side: `RETURN OLD` for DELETE, `RETURN NEW` everywhere else. `TG_TABLE_NAME` preserves the original quoted case for tables Prisma created with quoted identifiers — `"Card"` arrives at the client as the literal string `"Card"`, matching the TypeScript discriminator union without any case-folding.
- **Proxy matcher must exclude `/api/events`.** Same reasoning as `/api/trpc`: an EventSource can't follow a 307 HTML redirect to `/signin`. Without the exclusion an expired-session connection fails with an opaque `net::ERR_INVALID_RESPONSE` instead of a clean 401 JSON the client hook can react to. The SSE route owns its own `auth()` check; the proxy step is redundant.
- **Disable EventSource's built-in retry and roll your own backoff.** EventSource auto-reconnects with an interval the server can hint via the SSE `retry:` field, but the resulting cadence in dev (Fast Refresh, server restarts, autoPort changes) is hard to predict and feels flaky. `es.close()` inside `onerror` disables native retry; a `setTimeout(connect, delay)` with exponential backoff (1s → 2s → … → 30s cap) and `retry = 0` in `onopen` is predictable and survives a server bounce cleanly. Verified during Fast Refresh — the reconnected `EventSource` picked up subsequent NOTIFYs without intervention.
- **`runtime = "nodejs"` + `dynamic = "force-dynamic"` on the SSE route.** `pg.Client` is a TCP-via-Node-`net` implementation; the edge runtime has no `net`. Setting `runtime` explicitly future-proofs against a Next default change. `force-dynamic` blocks static optimization — a long-lived stream can't be cached. Both are one-line `export const` statements at the top of `app/api/events/route.ts`.
- **SSE chunks need `\n\n` separators, and `: ` prefixes the comment frame.** Two consecutive newlines terminate an event; a single `\n` leaves the parser holding the event in a buffer indefinitely. Heartbeats are `: ping\n\n` (the `:` makes them comments — clients ignore the content, but the bytes traverse intermediaries and reset idle timeouts). Data lines are `data: ${json}\n\n`. Two paths can both try to close the stream on disconnect — the `cancel` callback on the `ReadableStream` and the `req.signal.abort` listener — dedup with a `cleanedUp` boolean since `removeSubscriber` + `clearInterval` are idempotent and which path fires first depends on disconnect kind.

## Phase 9 (Auth.js v5 server actions, transactional deactivation, optimistic side-effects)

- **Sign-out from a `"use client"` component without `SessionProvider`.** Wrap `signOut({ redirectTo })` in a `"use server"` action in its own file (e.g. `app/(app)/_actions/signOut.ts`), import the action into the client header, and render `<form action={signOutAction}><button type="submit">Sign out</button></form>`. Auth.js v5 server actions posted through React's form `action` prop do NOT need a CSRF token (unlike a raw `<form action="/api/auth/signout" method="POST">`, which requires one). The whole pattern lets a parent RSC layout fetch `auth()` once and pass a narrow `{ name, role }` `viewer` prop to the client header — no `<SessionProvider>` needed just to render a name + role badge.
- **Inline role-change gate inside an `adminProcedure` mutation.** Instead of splitting `people.update` into `update` (admin) and `setRole` (manager), the single resolver does `if (patch.role !== undefined && ctx.role !== "MANAGER") throw new TRPCError({ code: "FORBIDDEN" })`. Cleaner public API surface, equivalent enforcement, and the UI can hide the role select for non-Managers as a parallel guard. Works because the patch payload encodes intent: if `role` is absent from the partial, the gate is moot.
- **Person deactivation must reassign active cards in the same transaction.** `cards.list` still returns cards whose `assigneeId` points at an inactive person, but `people.list` filters to active and the Board groups by person — the result is invisible orphans. Fix: `tx.card.updateMany({ where: { assigneeId: input.id, archivedAt: null }, data: { assigneeId: null } })` in front of the `person.update`, both inside `$transaction`. Archived cards keep their assignee on purpose, for History (Phase 10). Return the reassigned `count` so the UI can disclose it.
- **Contract deactivation needs no card surgery.** Asymmetric with Person on purpose: the Card → Contract relation is `onDelete: Restrict`, so the row never disappears, and `cards.list` joins Contract directly without an `active` filter. The chip on existing cards keeps rendering after deactivation — only the new-card picker + filter chips hide the contract. The "Contracts are a column dimension" instinct is wrong; here they're a relation only.
- **`useOptimisticListMutation` + an override `onSettled` for sibling caches.** The hook returns `{ onMutate, onError, onSettled }` keyed to a single list query. When the mutation also affects sibling queries (e.g. `people.deactivate` reassigns cards → `cards.list` is stale), spread the hook then override `onSettled` to also invalidate the siblings: `onSettled: () => { void utils.people.listAll.invalidate(); void utils.people.list.invalidate(); void utils.cards.list.invalidate(); }`. React Query dedupes overlapping invalidations into one refetch per key, so no extra round-trips even when SSE invalidates the same keys concurrently.
- **Self foot-gun protection on destructive admin actions is a UI concern.** The Access page disables Remove on the viewer's own allowlist row — a Manager removing themselves locks themselves out at next sign-in, recoverable only via DB access or `BOOTSTRAP_MANAGER_EMAIL` re-seed. Pattern: pass the resolved viewer identity (email here) as a prop from the RSC, compare in render, disable + add a `title` attribute explaining why. The server-side enforcement would need allowlist-CRUD to know the caller's email-vs-id (it does, via `ctx.userId` joined to Person.email), but at 4–8 users the UI guard is enough and avoids special-casing on the API surface.
- **`preview_fill` / `preview_click` quirks under React 19 controlled components.** Setting an `<input value>` via `el.value = "x"` does NOT trigger React's onChange — React patches the native setter and tracks the previous value via its own internals. Workaround: `Object.getOwnPropertyDescriptor(Object.getPrototypeOf(el), 'value').set.call(el, val); el.dispatchEvent(new Event('input', { bubbles: true }))`. Same shape works for `<select>` and `<textarea>`. Same root cause as the Phase 6/7 `preview_click`-on-Radix gotcha. Separately, clicking a button via `.click()` then immediately reading the React-driven DOM result can race the re-render — the click handler queues the state update but the read happens before commit. Either trust the click landed or queue a microtask before reading.

## Phase 10 (Prisma client extensions, JSONB filters, Radix DropdownMenu + Toast, AsyncLocalStorage)

- **Prisma client extensions can't cleanly participate in caller-opened `$transaction`s.** A `model.$allOperations` hook fires in whatever client context the operation was invoked (tx or base), but ancillary queries inside the hook — `client.auditLog.create(...)` to write the audit row, `client.findUnique(...)` to capture the before — execute on the BASE client, not the surrounding tx client. The audit insert lands on a fresh connection outside the transaction and won't roll back if the mutation aborts. Detecting "am I currently inside a tx" isn't exposed by the extension API. Pragmatic resolution: an explicit `writeAudit(tx, ...)` helper called inside each router's `$transaction`. The spec wanted a transparent extension; the explicit-call pattern is more invasive (one extra import + line per mutation) but actually achieves the spec's atomicity requirement, which the extension can't.
- **Prisma's `InputJsonValue` rejects `Date`.** Returns from `include`-laden queries (`include: { assignee, contract }`) contain Date objects (createdAt, dueDate, assignmentDate, archivedAt). Passing one directly to `tx.auditLog.create({ data: { before: rowWithDate } })` errors at runtime with "Type 'Date' is not assignable to type 'InputJsonValue'". Fix: `JSON.parse(JSON.stringify(value))` round-trip so Date → ISO string, undefined → dropped. The diff renderers on the read side accept either string or Date, so the round-trip is one-way.
- **`prisma migrate dev` doesn't regenerate the client for the editor session.** After `npx prisma migrate dev --name add_person_lastseenat` succeeded, tsc still threw "Property 'lastSeenAt' does not exist on type 'PersonSelect'". The migration SQL ran against Postgres, but the `.prisma/client/index.d.ts` file wasn't refreshed. `npx prisma generate` fixes it. `npm install` runs `prisma generate` via postinstall, but mid-development the migrate command alone is not enough. After any `schema.prisma` edit, follow with `prisma generate` (or restart `next dev`, which triggers a fresh generate via its own dev hooks).
- **Postgres JSONB path filters work in Prisma 7.** `where: { after: { path: ["assigneeId"], equals: userId } }` compiles to `("after"->'assigneeId')::text = $1` and lets the bell query "audit events where the user is the assignee on either side" without a JS-side filter or N+1 lookups. JsonNull-stored values silently don't match path queries, so `create` events (which have `before: null`) are excluded from the before-side OR clause without an explicit `NOT NULL` check — exactly what the unread-bell logic wants.
- **`AuditLog` inserts do NOT fire the Phase 8 NOTIFY trigger.** The trigger is registered on `Card`, `Person`, `Contract` only — AuditLog writes are silent. For production code this is fine because writeAudit() runs inside the same tx as the underlying Card mutation, which DOES fire NOTIFY → client invalidates audit.listForUser → toast pipeline picks it up. For testing the toast pipeline with a synthetic audit row, INSERT into AuditLog AND fire a no-op UPDATE on any Card (`UPDATE "Card" SET position = position WHERE ...`) — the UPDATE triggers NOTIFY which invalidates the audit query. Don't add a NOTIFY trigger to AuditLog itself: it would double-fire (mutation's own trigger + audit row's trigger) for the same logical change.
- **Radix DropdownMenu `Trigger` ignores synthetic `.click()`.** Same root cause as the Phase 6 Radix Dialog gotcha — Radix uses pointer events. `preview_click` on the bell button does nothing visible. Workaround in `preview_eval`: dispatch `new PointerEvent('pointerdown', { clientX, clientY, pointerId: 1, pointerType: 'mouse', bubbles: true })`, then `pointerup`, then `MouseEvent('click', { clientX, clientY, bubbles: true })` — all with the same coords. Real-user interaction works fine; the browser fires the full pointer sequence naturally.
- **Stacking two Radix Dialogs (CardEditModal + HistoryModal) works without explicit z-index management.** Both `Dialog.Portal`s mount to `document.body` in mount order; the upper one captures focus trap; Escape closes the upper first; click-outside-to-close on the upper one returns focus to the trigger inside the lower one. Each needs its own `Dialog.Root` with independent open state. No `nested-dialog` API or focus-handoff config required.
- **Toast pipeline needs a "seed seen set on first render" guard.** Without it, the very first `audit.listForUser` response (which contains every recent event affecting the viewer — up to 30 entries) dumps ~30 toasts on page load. The fix: an in-memory `seenRef = useRef<Set<string> | null>(null)` that on first non-null response gets seeded with all current IDs without firing any toasts. Subsequent responses diff against the seed and only toast genuinely new entries. Tradeoff: events that arrived while the page was closed land in the bell dropdown but don't pop a toast. Acceptable for an internal tool; matches Slack / Linear UX.
- **`prefs.markRead` invalidates `audit.unreadCount` only, NOT `audit.listForUser`.** Otherwise the dropdown contents empty out as the user is reading them (the lastSeenAt bump makes the next `listForUser` fetch return events newer than now() — nothing). The split between `unreadCount` (gated on lastSeenAt) and `listForUser` (last 30 events regardless) means the badge clears immediately on open while the user can still read what's in front of them.
- **`audit.listForUser` filter excludes self-actions at two layers.** Server-side: `where: { actorId: { not: ctx.userId } }`. Client-side: `viewerSummary(event, viewerId)` returns `null` when `event.actor?.id === viewerId`. Either alone would catch the common case, but both belt-and-suspenders so a system-actor event (actorId null) without a viewer-relative phrasing doesn't toast as a phantom notification.
- **Stale-content guard on date inputs round-trips through string conversion.** Saving an unedited card via the CardEditModal silently bumps `assignmentDate` / `dueDate` by ~1 calendar day in some timezones because the `<input type="date">` value (YYYY-MM-DD, no zone) becomes `new Date("2026-05-21")` (UTC midnight) on Save, which renders back in local time as the previous calendar day. Surfaced by Phase 10's audit-row diff showing "Assignment date: May 21 → May 20" for an unchanged-looking save. Not introduced by Phase 10 — the History view just exposed an existing CardEditModal date-handling quirk. Followup: clamp the input round-trip to noon-local or store as plain YYYY-MM-DD strings on the wire.

## Phase 11 (React 19 effect ordering, `useState` initializer, provider order, Auth.js v5 caching)

- **The "ThemeProvider in root + Bridge in (app)" split is a React 19 trap.** Intuition: have a `localStorage`-backed `ThemeProvider` in root (works for signed-out routes) and a `PreferencesBridge` in `(app)` that reads DB prefs server-side, then reconciles to ThemeProvider on mount via `setTheme(initial.theme)` (in `useLayoutEffect`) and writes back changes (in a `useEffect` keyed off `[theme, lastSyncedRef]`). What actually happens under React 19 concurrent rendering: the write-back `useEffect` from render 1 fires with render-1's stale theme closure ("soft" — the ThemeProvider default) BEFORE the `setTheme("bold")` from `useLayoutEffect` causes a re-render. The effect compares stale `theme === "soft"` against the ref I updated to `initial.theme === "bold"`, sees a mismatch, and queues a `prefs.set({theme: "soft"})` — overwriting the DB with the default state. Empirically reproduced with debug logs showing `queueWrite {"theme":"soft"}` firing first, then `queueWrite {"theme":"bold"}`, with debounce collapsing both into one POST of `{theme:"bold"}` (correct in this trace but wrong as soon as the initial DB value is "bold"). Fix: unify ownership in a single provider at root, initialize state directly from server-fetched `initial.theme`, no reconciliation step. The `useState(() => initial.theme)` initializer runs once with the authoritative value; write-back compares against `lastSyncedRef` initialized to the same value, so the initial render is a no-op even when StrictMode double-mounts.
- **Child effects fire BEFORE parent effects on initial mount.** Standard React behavior but easy to forget. A child component's `useEffect` runs first, then the parent's. So a `PreferencesBridge` (child of `ThemeProvider`) running reconciliation in `useEffect` runs *before* `ThemeProvider`'s `useEffect` that reads localStorage. Doesn't matter for the unified-provider design, but worth knowing — the obvious "do reconciliation in a child effect" pattern doesn't behave the way you might predict.
- **`useState(initializer)` runs the initializer ONLY ONCE per component instance.** `useState(() => initial.theme ?? DEFAULT_THEME)` is critical — the function form runs once at mount. Writing `useState(initial.theme ?? DEFAULT_THEME)` (without the wrapper) re-evaluates the expression every render, but React still only USES the value from the first call (subsequent renders' initializer arg is discarded). So both work in practice, but the function form is more honest about the one-shot nature, and matters when `initial` could change identity across renders.
- **`TRPCProvider` must wrap any provider that calls tRPC hooks.** Reordering providers at root is a 5-second edit that erases on a refactor. Pre-Phase-11 ordering was `<ThemeProvider><TRPCProvider>{children}</TRPCProvider></ThemeProvider>` because ThemeProvider didn't need tRPC. Phase 11's new `PreferencesProvider` calls `trpc.prefs.set.useMutation()` and needs the inverse: `<TRPCProvider><PreferencesProvider>{children}</PreferencesProvider></TRPCProvider>`. Symptom of the wrong order: `Unable to find tRPC Context. Did you forget to wrap your App inside withTRPC HoC?` thrown at render. The hint that "withTRPC" is mentioned is misleading for tRPC v11 + App Router (which uses `<TRPCProvider>`); same root cause though.
- **Auth.js v5 `auth()` is memoized per request via React.cache, so duplicate calls across layouts are free.** Root layout calling `auth()` (to decide whether to fetch prefs) and `(app)/layout` calling `auth()` (to build the viewer prop) used to feel like two decryptions; under v5 they share a single execution per server-rendered request. So you can safely call `auth()` wherever you need it in RSC code without threading the session through props. Verified by checking the absence of duplicate decryption work in dev logs after the duplicate call landed.
- **React StrictMode dev double-mount does NOT cause duplicate DB writes when write-back effects are properly idempotent.** Each StrictMode "remount" creates fresh refs (lastSyncedRef re-initialized to `initial.theme`), but state is also re-initialized to the same value, so `theme === lastSyncedRef.current` and the write-back skips. Tested: 0 prefs.set POSTs on reload with matching state, even with React StrictMode active. (Naively, you might expect StrictMode-induced double-mount to cause every effect to fire twice and each fire to write — only true if the comparison check is wrong.)
- **`window.location.reload()` is a soft reload — `performance.getEntriesByType('resource')` resets cleanly.** Doesn't carry over the old page's resource entries. Useful for verifying "0 POSTs since this reload" tests without needing a fresh tab. The entry buffer does have a finite size though, so don't use it for "ever fired" assertions — only "since the most recent navigation".
- **The pre-paint script + state initializer pair lets the DB be authoritative across devices with at most a one-frame flicker.** Pre-paint reads localStorage (might be stale relative to DB), sets data-theme. React renders with state initialized from server-fetched DB value (authoritative). The `useEffect` that writes data-theme corrects the DOM within one frame if they differ. The flicker is the floor for cross-device sync without a server-side render-with-cookie scheme; acceptable for an internal tool where most reloads agree.
- **Cross-device live sync of preferences is NOT wired up (deliberate omission).** Phase 11 doesn't add a NOTIFY trigger on `UserPreference`, so device B doesn't see device A's toggle until next sign-in / next manual `prefs.get` fetch. The spec only required "carry over on sign-in" which works fine via the root-layout server fetch. Adding live sync would mean a new trigger + adding `UserPreference` to the SSE invalidation key map in `useRealtimeSync` — possible but unneeded for the 4-8 user use case.

## Phase 12 (Next.js standalone in Docker, Prisma 7 build-time env, Caddy SSE, compose env strictness)

- **`prisma generate` inside the builder stage fails once prisma.config.ts is on disk.** The config file calls `env("DATABASE_URL")` and `prisma/config`'s `env()` helper resolves strictly at config-load time — `prisma generate` chokes loading the config even though generation doesn't actually need a DB connection. Surfaces as `PrismaConfigEnvError: Cannot resolve environment variable: DATABASE_URL`. Fix: don't re-run `prisma generate` in the builder. The `deps` stage's `npm ci` postinstall already generates the client BEFORE prisma.config.ts is copied in (so Prisma falls back to defaults that don't require the URL), and `COPY --from=deps /app/node_modules` carries `.prisma/client` into the builder. `next build`'s trace step then pulls the generated client into `.next/standalone` automatically.
- **`next build` evaluates `lib/db.ts` during page-data collection.** Any API route that imports the Prisma client transitively (e.g. `auth.ts` → `lib/db.ts`) gets loaded for module-level metadata extraction. If `lib/db.ts` validates `DATABASE_URL` eagerly at construction time (we do — single `createClient()` call on import), the build fails with `Failed to collect page data for /api/auth/[...nextauth]`. Fix: an `ENV DATABASE_URL=postgresql://build:build@localhost:5432/build` in the builder stage. The URL is never used to make a connection (every app route is `force-dynamic` and instrumentation's `bootstrap()` swallows DB errors) — it just satisfies the module-load validation. Real URL comes from compose at runtime.
- **`output: "standalone"` requires a `public/` directory to exist even when empty.** The standard `COPY --from=builder /app/public ./public` in the runner stage fails with `"/app/public": not found` on repos that don't ship static assets. Fix: `RUN mkdir -p public` in the builder before any COPY. Cheaper than carrying a `public/.gitkeep` placeholder, and keeps the Dockerfile portable across repos.
- **Stage ordering matters for the default build target.** `docker build .` without `--target` builds the LAST stage. Put `runner` last so a bare `docker build` produces the app image; put `migrator` before it (compose explicitly targets `migrator`, so its position doesn't matter for compose). The alternative — making `migrator` last — means manual `docker build` would produce the wrong default image.
- **Migrator stage needs the full dev deps to load `prisma.config.ts`.** Prisma 7's CLI loads `.ts` configs via jiti (no separate tsx needed at runtime), but the config file itself imports `dotenv` for `.env`-loading convenience in dev. `dotenv` is a devDep; `--omit=dev` strips it; loading fails. Simplest fix: copy node_modules from a `deps` stage that ran `npm ci` without `--omit=dev`. The resulting migrator image is ~320 MB but it's a one-shot that exits after `migrate deploy`, so the size is OK.
- **Prisma 7 driver-adapter eliminates the alpine/musl headache.** With `PrismaPg` providing the runtime, Prisma 5/6's native Rust binary engine + `binaryTargets = ["native", "linux-musl-arm64-openssl-3.0.x"]` dance is gone. node:22-alpine works as both builder and runner. Saves an entire class of "engine not found" debugging that used to plague containerized Prisma deploys.
- **Compose `${VAR:?msg}` syntax hard-fails at interpolation time, not at runtime.** `docker compose config` / `up` parses the YAML, substitutes env vars, and errors out the moment it hits an unset required variable. Empty string counts as unset (per POSIX `:?`). This is the right behavior — it surfaces a missing `NEXTAUTH_SECRET` immediately rather than failing on the first sign-in attempt after a successful boot. The `.env.production.example` deliberately ships with blank values so a `docker compose -f docker-compose.prod.yml --env-file .env.production.example config` errors out as a sanity check.
- **Caddy's `encode` directive at the site-block level applies to ALL responses — including SSE.** Putting `encode zstd gzip` at the top of the site block and then carving out `@sse handle` for the events endpoint does NOT exempt SSE from compression — the encoder still wraps the response. Compression buffers chunks until the encoder flushes, defeating the entire real-time pipe. Fix: move `encode` INSIDE the catch-all `handle { ... }` block; the `@sse handle { reverse_proxy app:3000 { flush_interval -1 } }` block then runs WITHOUT encode. The two handle blocks are mutually exclusive (first match wins), so SSE bypasses the encoder cleanly.
- **`flush_interval -1` is per-`reverse_proxy` and disables Caddy's response buffering.** Without it, even an uncompressed SSE response can sit in Caddy's internal write buffer until it fills. `-1` flushes after every write (each `data: ...\n\n` chunk forwarded immediately). Pair with `transport http { response_header_timeout 0 }` so Caddy doesn't time out on the long-lived stream.
- **The `caddy_data` volume holds the ACME account key + issued certs — losing it forces re-issuance.** Let's Encrypt rate-limits duplicate certs to 5/week. A naive `docker compose down -v` (with `-v` to remove volumes) on a stable prod stack burns the ACME account and one of the 5 reissuance budget for the week. Cite this in the README and never document `down -v` in operational commands — `down` (without `-v`) is the right verb for restarts.
- **Cross-platform line endings: `*.sh eol=lf` in `.gitattributes`.** Without it, Windows git's autocrlf may rewrite shell scripts with CRLF — the resulting `\r` after `#!/usr/bin/env bash` breaks the shebang on Linux deploy with cryptic `/usr/bin/env: 'bash\r': No such file or directory`. The fix is one `*.sh text eol=lf` line in `.gitattributes` plus extending to `Dockerfile`, `Caddyfile`, and `*.yml` for the same reason. Mark scripts executable in the git index with `git update-index --chmod=+x` (Windows git doesn't track exec bit via filesystem).
- **Add `/api/health` to the proxy matcher exclusion list.** Otherwise the Auth.js v5 proxy wraps the health request, sees no session, and 307s to `/signin` — Docker's healthcheck reads the redirect as a non-2xx and marks the container unhealthy. Same reasoning as `/api/trpc`, `/api/events`, `/api/auth`: routes that need their own auth contract (or none, in this case) belong outside the proxy.
